# epub2md — design

A Rust CLI that converts an epub into a single YAML-headed markdown file plus an `images/` subdirectory, optimized for token-efficient reading by agents.

## Goals

- One markdown file per book. Books are immutable, so byte and line offsets in the YAML frontmatter are stable seek targets.
- Every transform is deterministic. No agentic judgment, no LLM calls, no network. Same input → byte-identical output.
- Use a real epub library and a real HTML parser. We write only the things that have to be project-specific: the HTML→Markdown walker, the offset table, and the YAML header.

## Non-goals

- No HTML cleanup that requires judgment (e.g., "drop ornamental images").
- No table width handling, no line wrapping, no reflow.
- No conversion of non-epub formats.
- No extraction or summarization step. This tool emits a faithful structural translation of the book.

## CLI

```
epub2md <input.epub> [--output-dir output] [--force]
```

- Refuses to overwrite an existing output directory unless `--force`.
- Exits non-zero with a descriptive message on any structural problem with the input.
- No other flags. Behavior is fixed.

## Output layout

```
<output-dir>/<slug>/book.md
<output-dir>/<slug>/images/<basename>
```

`<slug>` is derived from the epub's OPF metadata (`title` + first `creator`, slugified). If metadata is missing, falls back to slugifying the input filename.

## Frontmatter

```yaml
---
title: How to Take Smart Notes
authors: [Sönke Ahrens]
publisher: Sönke Ahrens
published: 2017
isbn: 978-1542866507
language: en
source_file: How to Take Smart Notes - Sönke Ahrens.epub
chapters:
  - title: Introduction
    line:           14
    byte:          412
  - title: Chapter 1
    line:          261
    byte:        18844
---
```

- `chapters[].line` and `chapters[].byte` are absolute positions in `book.md` (frontmatter included). Lines are 1-indexed; bytes are 0-indexed.
- Every numeric offset is left-padded with spaces to a fixed 10-character field. YAML plain scalars discard leading separator whitespace, so consumers parse them as integers. Leading-pad (not trailing-pad) is chosen because trailing whitespace is routinely stripped by editors and pre-commit hooks.
- Every spine entry produces one chapter row, including title pages, copyright pages, and indexes. Per requirements, all text is preserved.
- Cover image is omitted entirely (not copied, not referenced).

## Implementation

### Language and crates

Rust. Single static binary.

- `epub` — reads container.xml, OPF manifest, spine, metadata, raw resource bytes.
- `scraper` (built on `html5ever`) — parses each spine document into a real DOM.
- `serde` + `serde_yaml` — YAML frontmatter emission.
- `clap` — argument parsing.
- `slug` — slugifies titles.
- `anyhow` / `thiserror` — error handling.

### Pipeline

Five modules, each one stage of the pipeline:

1. **`load`** — open the epub, read metadata, resolve spine order, build a map of all resources keyed by manifest path.
2. **`extract`** — for each spine document, parse HTML and walk the DOM into an in-memory `Block` tree. One `Document` per spine entry.
3. **`assemble`** — concatenate documents in spine order. Promote each spine doc's resolved title to an H2 chapter heading. Collect footnote definitions per chapter and inline them at chapter end.
4. **`render`** — serialize the `Block` tree to GFM markdown into a `String`. Record body-relative byte and line position for each chapter heading.
5. **`write`** — build the frontmatter (with absolute offsets), write `book.md`, copy referenced images into `images/`.

### Data model

The `Block` tree is the only intermediate representation. HTML in, `Block` out, markdown out — no regex on raw HTML.

```rust
enum Block {
    Heading { level: u8, text: Inline },
    Paragraph(Inline),
    BlockQuote(Vec<Block>),
    List { ordered: bool, items: Vec<Vec<Block>> },
    Table { header: Vec<Inline>, rows: Vec<Vec<Inline>> },
    CodeBlock { lang: Option<String>, code: String },
    Image { src: String, alt: String, title: Option<String> },
    HorizontalRule,
    FootnoteDef { id: String, content: Vec<Block> },
}

enum Inline {
    Text(String),
    Emphasis(Vec<Inline>),
    Strong(Vec<Inline>),
    Code(String),
    Link { href: String, children: Vec<Inline> },
    Image { src: String, alt: String, title: Option<String> },
    FootnoteRef(String),
    LineBreak,
    Concat(Vec<Inline>),
}
```

### HTML → Block mapping

| HTML | Block |
|---|---|
| `<h1>`–`<h6>` | `Heading` at that level |
| `<p>` | `Paragraph` (CSS classes ignored) |
| `<blockquote>` | `BlockQuote` |
| `<ul>` / `<ol>` / `<li>` | `List` / list item |
| `<table>` and children | `Table` |
| `<pre>` (or `<pre><code>`) | `CodeBlock` |
| `<code>` (inline) | `Inline::Code` |
| `<img>` block-level | `Block::Image`; otherwise `Inline::Image` |
| `<em>` / `<i>` | `Inline::Emphasis` |
| `<strong>` / `<b>` | `Inline::Strong` |
| `<a href>` | `Inline::Link` |
| `<br>` | `Inline::LineBreak` |
| `<hr>` | `HorizontalRule` |
| `<sup><a href="#fnX">…</a></sup>` or `epub:type="noteref"` | `Inline::FootnoteRef("X")` |
| `<aside epub:type="footnote">` or `<div class="footnote">` | `FootnoteDef` (hoisted out of flow) |
| `<span>`, `<div>`, unknown tags | transparent — recurse into children |
| Any element with no text content and no non-empty children | dropped |

### Chapter title resolution (deterministic)

For each spine entry, in priority order:

1. The matching label in `<nav epub:type="toc">` (epub3) or `toc.ncx` `navPoint` (epub2).
2. The spine document's `<title>` element.
3. The first `<h1>` or `<h2>` text in the document.
4. `Untitled (<spine-filename>)` as a final fallback.

### Footnote handling

- Inline `[^cN-id]` for references, where `N` is the 1-based chapter index and `id` is the original footnote id from the epub. Namespacing prevents cross-chapter collision.
- Definitions emitted at the end of each chapter, beneath an `---`, in reference order.

### Markdown serialization rules

Hand-written serializer. No `pulldown-cmark` round-trip — pulldown is a parser, and existing renderers add opinions we don't want.

- **Headings**: chapter starts at `##` (level 2). In-chapter headings shift down one level (`<h1>` → `##`, `<h2>` → `###`, etc.).
- **Paragraphs**: blank line above and below. Internal whitespace collapsed to single spaces. **No line wrapping** — one paragraph is one line. Wrapping would break offsets across diffs and offers no value to a reader that doesn't eyeball.
- **Bold/italic**: `**…**`, `*…*`. Code: backticks; if content contains backticks, use the longest run + 1.
- **Links**: `[text](href)`. Internal cross-spine links rewritten to anchor links within `book.md`. Anchors are deterministic: chapter heading → `chapter-N`; in-document `id="foo"` → `chapter-N-foo`.
- **Anchor emission**: for an in-document `id="foo"` that is not on a heading, emit `<a id="chapter-N-foo"></a>` on its own line immediately before the markdown rendering of the element that bore the id. Narrow allowed use of HTML in markdown.
- **Lists**: `-` for unordered, `1.` `2.` `3.` for ordered. Nested lists indented by 2 spaces.
- **Block quotes**: `> ` prefix on each line, recursive.
- **Code blocks**: triple-backtick fenced, with language if `<code class="language-foo">`, else bare fence.
- **Tables**: GFM pipe tables. `|` inside cells escaped as `\|`. Newlines inside cells replaced with `<br>` (only structural place we use HTML in markdown — GFM has no other option).
- **Images**: `![alt](images/<basename>)` on its own paragraph if block-level, inline otherwise. `title` becomes `![alt](src "title")`.
- **Horizontal rule**: `---` on its own line.

### Offset construction (single-pass)

Frontmatter byte size is known up front because the chapter list, titles, and metadata are all available before body rendering. The numeric values are leading-padded to a fixed 10-digit field, so changes to the digit count do not change the frontmatter byte size.

1. Build the frontmatter with placeholder offsets (`byte: 0` padded). Measure its byte size and line count.
2. Render the body once. While rendering, capture each chapter's body-relative byte and line position.
3. Add the frontmatter byte size and line count to each chapter's offsets to get absolute file positions.
4. Rewrite the placeholder offsets with the real values, still padded to the same width.
5. Concatenate frontmatter + body. Write.

Hard error if any offset exceeds 10 digits (would require a > 10 GB book — does not occur in practice).

### Image handling

- Every image referenced from a spine document is copied to `<output>/<slug>/images/<basename>`.
- Cover image (per OPF `<meta name="cover">` or `properties="cover-image"`) is skipped.
- Basename collisions across different OPF directories resolved by suffix counter — `foo.jpg`, `foo-2.jpg`, `foo-3.jpg` — assigned in sorted manifest path order. References are rewritten to match.
- Hard error if a spine doc references an image that the manifest does not contain.

### Determinism

- Same input epub → byte-identical output.
- Sorted manifest order for image collision resolution.
- No timestamps, no host metadata, no pretty-printing variability.

## Errors and edge cases

### Hard errors (non-zero exit)

- Input missing, not a zip, or fails epub container/OPF parsing.
- Spine references a manifest entry that does not exist.
- Spine document fails HTML5 parse.
- Output directory exists and `--force` not passed.
- Image referenced from a spine doc but missing from the manifest.
- Any chapter offset exceeds 10 digits.

### Handled edge cases

- **Cover-only spine documents**: kept as chapter entries; if the rendered body is empty, the chapter heading is still emitted with a blank line so offsets remain meaningful.
- **Multi-file logical chapters**: not merged. Each spine document is one chapter row.
- **External links** (`http`, `https`, `mailto`): kept verbatim.
- **Orphan footnote refs or defs**: emitted regardless; a warning is written to stderr.
- **Empty `<p/>`, `<em/>`, etc.**: dropped.
- **Whitespace**: collapsed within paragraphs; preserved inside `<pre>`.
- **Non-UTF-8 spine docs**: epub spec mandates UTF-8/UTF-16. The `epub` crate handles decoding; a decode error is a hard fail.

## Testing

All test fixtures are synthetic. No copyrighted material — neither input epubs nor derived markdown — is ever committed to this repository.

- **Unit tests** per module: HTML-fragment-to-Block, Block-to-markdown, offset math, slug derivation, image collision resolver.
- **Snapshot tests** (`insta`) on the markdown output of small synthetic epubs we build in-test. Fixtures cover paragraphs, lists, tables, code, footnotes, images, nested structures, and the listed edge cases. Snapshot diffs surface any output drift.
- **Hand-authored fixture epubs** committed under `tests/fixtures/`. Each one targets a specific shape (footnote-heavy, table-heavy, multi-spine cover-only, etc.), built from public-domain or original prose written for the test.
- **Determinism test**: run the converter twice on the same input, byte-compare outputs.
- **Offset verification test**: parse the emitted YAML, then for each chapter seek to `byte` in the file and assert the next non-whitespace bytes are `## <title>`. Same for `line` against a line-indexed read.
- **Ad-hoc validation against real books** is supported but outputs are never committed. The `.gitignore` excludes `*.epub` and any output directory that contains converted books. CI does not run the tool against unauthorized content.
